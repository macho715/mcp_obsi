### What is now correct

`CONSOLIDATED-00` is now a valid canonical spine. It explicitly bans `assignedFlowCode` in Port, `extractedFlowCode` in Document/OCR, `costByFlowCode` in Cost, and `Flow Code 3/4/5` language in Marine/Bulk, while reserving Flow Code to `WarehouseHandlingProfile.confirmedFlowCode` only. That is the correct ontology boundary. 

`CONSOLIDATED-06` is the strongest extension after the rebuild. It now ties `confirmedFlowCode` to **warehouse-only handling logic**, triggered only after **M110 (Warehouse Received)**, and defines the warehouse-local code set inside `WarehouseHandlingProfile` rather than as a shipment lifecycle code.  

`CONSOLIDATED-08` is still correctly isolated as an **Evidence / Communication Layer**, not part of the core logistics semantic spine. That separation is correct and should be preserved. 

## The main blocker now

The current corpus contains **two incompatible meanings for the same 0–5 code range**.

In the revised warehouse/material-handling logic, `confirmedFlowCode 0~5` is now a **warehouse storage / handling classification** such as `Standard Inside`, `Standard Outside`, `Special Inside`, `Special Outside`, `Hazmat`, and `OOG / Abnormal`. But `CONSOLIDATED-02` still describes Flow Code 0–5 as a **route algorithm** tied to warehouse hops, MOSB transit, AGI/DAS rules, and KPI logic. `CONSOLIDATED-09` also still treats Flow Code 1/2/3/4/5 as shipment-route KPI buckets. That means the same numeric code set now means two different things in the same ontology family. This is a hard publish blocker.

`CONSOLIDATED-00` is **approved** as the canonical authority. Keep it unchanged and treat it as the only source of truth for terminology, RoutingPattern, Milestone, and Flow Code ownership. 

`CONSOLIDATED-06` is **approve with minor cleanup**. Its warehouse section is now structurally correct: `confirmedFlowCode` is generated after M110 and held inside `WarehouseHandlingProfile`. This is the right pattern and should be the template for the remaining extensions. 

`CONSOLIDATED-03` is **conditional only**. Its domain rule now correctly says documents should extract `routeEvidence` and `destinationEvidence` and should not assign `confirmedFlowCode`. `CONSOLIDATED-07` is **not approved**. `CONSOLIDATED-04` is **not approved**. It now uses `route_type` wording in parts, but the file still says Marine/Bulk operates in “Flow Code 3 and 4,” still calls MOSB the **Flow Code anchor**, and still defines bulk-cargo route typing with an integer-coded property even though the intended values are route strings. `CONSOLIDATED-05` is **not approved**. Its top domain rule is correct: Cost should use `route_type` and `WarehouseHandlingProfile` evidence as cost drivers, and it must not own `confirmedFlowCode`. But the body immediately contradicts that rule by saying **“Flow Code directly impacts logistics costs”** and by stating that Flow Code is extracted from invoice OCR and used for cost validation. That contradiction must be removed before publication.  

`CONSOLIDATED-02` is **not approved**. Its top-level governance now says the right thing, but the file still preserves a legacy **Flow Code Algorithm Ontology** as a 6-step route/KPI system and still treats that as a core warehouse artifact. That is incompatible with the revised warehouse-only meaning now used in `CONSOLIDATED-06`.  

`CONSOLIDATED-09` is **not approved**. It is internally mixed. Its governance header is correct, and some newer sections use `hasRouteType`, but the file still frames KPI logic as “Flow Code” analytics and older same-day revisions still expose `hasLogisticsFlowCode` as a case-level route property. That means the migration is incomplete and query semantics are unstable.   

`CONSOLIDATED-01` is **not the blocker**. The retrieved 2026-04-11 chunks show its governance is aligned with the spine. I would keep it in scope for a final example/payload scrub only after the extension migration is finished. 

## Exact patch sequence

1. **Freeze `CONSOLIDATED-00` as the non-negotiable authority.** No extension may redefine Flow Code outside `WarehouseHandlingProfile`. 

2. **Delete all remaining cross-domain Flow Code semantics** from `02`, `03`, `04`, `05`, `07`, and `09`.
