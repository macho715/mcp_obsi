## Section: Namespace + RDF Predicate + 호환 매핑 원칙

### 1. Namespace 구조

#### 1.1 추천 구조
스키마와 데이터 인스턴스를 분리합니다.

- `hvdc:`  
  `https://hvdc.logistics/ontology/core/`
- `hvdce:`  
  `https://hvdc.logistics/ontology/event/`
- `hvdco:`  
  `https://hvdc.logistics/ontology/ops/`
- `hvdcd:`  
  `https://hvdc.logistics/ontology/document/`
- `hvdck:`  
  `https://hvdc.logistics/ontology/knowledge/`
- `hvdcm:`  
  `https://hvdc.logistics/ontology/mapping/`

인스턴스는 별도 base를 둡니다.

- `res:`  
  `https://hvdc.logistics/resource/`

즉,
- 클래스/속성 = `ontology/...`
- 실제 shipment/case/event/invoice = `resource/...`

이렇게 분리하면 schema와 data를 같이 유지해도 충돌이 적습니다.

### 1.2 왜 이 구조가 좋은가
- `Shipment`, `Case`, `Event`, `Rule`, `Lesson`이 한 namespace에 섞이지 않습니다.
- `CONSOLIDATED-04` marine/LCT 호환도 `ops` 계층에서 처리하기 쉽습니다.
- 나중에 SHACL, JSON-LD context, mapping table을 붙이기 편합니다.

### 4. Compatibility Mapping 원칙

#### 4.1 의미 호환 우선
기존 `CONSOLIDATED-00~09`의 **이름을 그대로 복제하는 것**이 아니라,  
**의미가 어떻게 대응되는지**를 매핑표로 남깁니다.

예:
- 새 `OperationCarrier`  
  -> 기존 `debulk:Vessel` 또는 marine operation 개념과 의미 연결
- 새 `LCTOperation`  
  -> 기존 `MarineRoutingPattern`, `offshoreDeliveryPattern`, `OperationTask`와 연결
- 새 `IncidentLesson`  
  -> 기존 operations / communication / evidence 문서의 사례 개념과 연결

#### 4.2 one-to-many 허용
기존 문서의 개념 하나가 새 TTL에서는 두 개 이상으로 분해될 수 있습니다.

예:
- 기존 `vessel movement`
  - 새 TTL에서는
    - `OperationCarrier`
    - `PortCall`
    - `ArrivalEvent`
    - `DepartureEvent`
  로 분리될 수 있습니다.

#### 4.3 WhatsApp/Guide는 core가 아니라 knowledge layer
GUIDE 문서와 WhatsApp lesson은
- `Shipment`
- `Case`
- `Invoice`
같은 core class를 직접 바꾸지 않습니다.

대신
- `OperatingRule`
- `IncidentLesson`
- `CommunicationEvidence`
- `RecurringPattern`
으로 들어가고,
core class와는 link만 맺습니다.

#### 4.4 CONSOLIDATED-04 특별 원칙
`CONSOLIDATED-04-barge-bulk-cargo.md`와는 아래 4개를 우선 맞춥니다.

- marine routing 의미
- MOSB staging 의미
- LCT/barge transport 의미
- port/jetty/anchorage operational location 의미

즉, `JPT-reconciled_v6.0.xlsx`는 새 TTL에서 `CONSOLIDATED-04`와 가장 강하게 매핑됩니다.

#### 5.2 `relatedToLesson` 같은 projection-friendly property
정식 canonical predicate와 별개로,
read model용 summary link를 만들 수도 있습니다.

예:
- canonical:
  - `IncidentLesson relatedToShipment Shipment`

  - `Shipment relatedToLesson IncidentLesson`

이건 query와 UI를 빠르게 해 줍니다.

#### 5.3 `confidenceScore`

예:
- `hvdck:confidenceScore`
- `hvdcd:attributionConfidence`

이렇게 하면 자동 연결과 수동 검증 대상을 구분하기 쉽습니다.

### 6. 추천 결론
최종 추천은 이렇습니다.

- namespace는 `core / event / ops / document / knowledge / mapping`으로 분리
- instance URI는 `resource/`로 별도 분리
- predicate는 동사형 object property와 명사형 data property로 통일
- 기존 `CONSOLIDATED-00~09`는 이름 복제가 아니라 의미 매핑으로 호환
- WhatsApp/analyses는 `knowledge` 계층에서 lesson/rule/evidence로 연결

이 구조면  
TTL을 전면 재설계해도 기존 문서군과 부딪히지 않고,  
나중에 dashboard와 Obsidian wiki까지 같이 연결하기 좋습니다.
