# `llm-wiki-karpathy` 상세 분석 보고서

## 1. 개요

이 레포는 “문서를 매번 다시 검색하는 방식”보다, 한 번 읽은 소스를 지속형 위키로 축적하는 방식을 구현한 개인 지식베이스 템플릿이다. 핵심 아이디어는 LLM이 원문을 읽고, 그 결과를 `wiki/` 아래의 상호연결된 문서 집합으로 재구성해 시간이 갈수록 더 풍부한 지식 네트워크를 만든다는 점이다. README는 이를 RAG의 대안이라기보다, **지식이 누적·정규화·연결되는 운영 패턴**으로 설명한다. ([GitHub][1])

즉, 이 레포의 본질은 “질문할 때마다 원문을 읽는 시스템”이 아니라, **원문 → 요약 → 개체 추출 → 링크 생성 → 분석 축적**의 장기 기억형 워크플로를 제공하는 데 있다. 소스가 늘어날수록 위키의 연결성과 분석 품질이 누적된다는 점이 이 프로젝트의 가장 큰 설계 특징이다. ([GitHub][1])

## 2. 이 레포가 해결하려는 문제

일반적인 LLM 활용은 대화 세션 단위로 휘발되기 쉽다. 문서 몇 개를 넣어 요약받는 것은 가능하지만, 다음 세션에서 그 맥락과 연결 구조가 쉽게 사라진다. 이 레포는 그 한계를 해결하기 위해, 대화 결과를 계속 저장 가능한 위키 문서로 변환하고, 그 위키를 다시 다음 질문의 기반으로 사용하는 구조를 취한다. ([GitHub][1])

따라서 이 프로젝트가 겨냥하는 문제는 다음 세 가지로 정리된다.

| 항목        | 의미                 | 근거                                                                       |
| --------- | ------------------ | ------------------------------------------------------------------------ |
| 지식의 휘발성   | 채팅 결과가 축적되지 않음     | README는 질문 결과를 분석 페이지로 저장해 지식이 누적된다고 설명함. ([GitHub][1])                  |
| 원문 재탐색 비용 | 매 질문마다 원문을 다시 뒤짐   | README는 원문을 한 번 읽고 persistent, interlinked wiki를 만든다고 설명함. ([GitHub][1]) |
| 연결 구조 부족  | 개념·제품·페르소나·분석이 분절됨 | 다양한 page type과 cross-reference 구조를 정의함. ([GitHub][1])                    |

## 3. 핵심 동작 방식

### 3.1 기본 메커니즘

운영 흐름은 매우 단순하다.

1. 사용자가 원문 파일을 `raw/` 폴더에 넣는다.
2. 에이전트에게 `ingest raw/파일명` 형태로 지시한다.
3. 에이전트가 문서를 읽고 핵심 내용을 논의한다.
4. `wiki/sources/`에 소스 요약 페이지를 만든다.
5. 제품, 기능, 페르소나, 개념 같은 개체를 새 문서로 생성한다.
6. glossary, index, overview, log를 갱신한다. ([GitHub][1])

즉, 단순 요약이 아니라 **문서 원문을 구조화된 위키 객체 집합으로 변환하는 ingest 파이프라인**이다. README는 한 개 소스가 5개에서 15개 정도의 위키 페이지를 갱신할 수 있다고 설명한다. 이는 입력 1건이 다수의 지식 노드와 링크를 만들어내는 그래프 지향 구조임을 뜻한다. ([GitHub][1])

### 3.2 질문 응답 메커니즘

소스를 넣고 위키가 쌓인 뒤에는 사용자가 위키 전체를 대상으로 질문할 수 있다. 예시로 README는 “모든 소스에서 식별된 주요 리스크는 무엇인가?” 같은 질문을 제시한다. 이때 에이전트는 원문 전체를 즉시 다시 훑는 것이 아니라, 이미 구조화된 위키를 읽고 종합 답변을 만든다. 그리고 그 답변을 새로운 위키 페이지로 저장할지 다시 묻는다. ([GitHub][1])

이 구조는 매우 중요하다. 질문이 끝나는 순간 답변이 사라지는 것이 아니라, **질문 자체가 새로운 분석 아티팩트가 되어 지식베이스를 확장**한다. 따라서 시간이 지날수록 이 시스템은 “문서 저장소”보다 “분석 축적 엔진”에 가까워진다. ([GitHub][1])

### 3.3 유지보수 메커니즘

README는 일정 횟수의 ingest 후 `lint the wiki`를 실행하라고 권장한다. 이 lint 단계에서 에이전트는 다음 문제를 점검한다.

* 페이지 간 상충
* 오래된 주장
* 고아 페이지
* 빠진 상호참조
* 용어 불일치 ([GitHub][1])

즉, 이 레포는 단순 생성형 시스템이 아니라 **지식 정합성 유지 단계**까지 포함한다. 운영 관점에서 보면 “데이터 수집 → 구조화 → 질의응답 → 정합성 점검”의 폐루프가 있다. ([GitHub][1])

## 4. 레포 구조 분석

README에 제시된 구조는 이 프로젝트의 설계 철학을 직접 보여준다. 핵심은 세 레이어 분리다. `raw/`는 원문 보관, `wiki/`는 AI가 관리하는 구조화 지식, `CLAUDE.md`는 AI가 따라야 하는 스키마다. ([GitHub][1])

### 4.1 폴더별 의미

* `raw/`: 사용자가 넣는 원문 저장소다. AI는 읽기만 하고 수정하지 않는다. 원문 불변성이 유지된다. ([GitHub][1])
* `wiki/`: AI가 생성·갱신하는 지식 레이어다. index, overview, glossary, log가 포함된다. ([GitHub][1])
* `CLAUDE.md`: 위키 구조, 페이지 포맷, 워크플로, 규칙을 담은 운영 매뉴얼이다. 다른 에이전트를 쓸 경우 이 내용을 컨텍스트에 넣으라고 안내한다. ([GitHub][1])
* `.obsidian/`: Obsidian에서 즉시 탐색할 수 있도록 준비된 설정 폴더다. ([GitHub][1])

### 4.2 구조적 장점

이 구조는 원문과 가공물을 명확히 분리한다. 실무적으로 보면 상당히 좋은 설계다. 원문은 변경 금지, 위키는 AI 관리, 스키마는 사람이 통제하는 식으로 책임 경계가 선명하다. 이는 감사 가능성, 추적성, 복구 가능성 측면에서 유리하다. README도 각 레이어의 소유권을 분명히 구분한다. ([GitHub][1])

## 5. 위키 페이지 타입 분석

이 레포는 단일 문서 요약 저장소가 아니라, 개체별 문서 유형을 구분한다. README에 따르면 AI는 상황에 따라 다음 유형의 페이지를 만든다. ([GitHub][1])

| 타입         | 저장 위치            | 역할                      |               |
| ---------- | ---------------- | ----------------------- | ------------- |
| Source     | `wiki/sources/`  | 원문 문서의 요약, 핵심 사실, 메타데이터 |               |
| Feature    | `wiki/features/` | 기능 단위 설명                |               |
| Product    | `wiki/products/` | 제품/도구 단위 설명             |               |
| Persona    | `wiki/personas/` | 사용자 유형, 목표, pain point  |               |
| Concept    | `wiki/concepts/` | 도메인 개념, 정의, 오해          |               |
| Style Rule | `wiki/style/`    | 글쓰기 규칙                  |               |
| Analysis   | `wiki/analyses/` | 비교, 갭 분석, 개요 같은 합성 산출물  | ([GitHub][1]) |

이 설계는 중요한 시사점을 가진다. 단순한 벡터 검색 시스템은 보통 문단 조각을 중심으로 검색하지만, 이 레포는 **개체 중심 knowledge modeling**을 취한다. 다시 말해, “문서”보다 “문서에서 추출되는 의미 객체”가 관리 단위다. 따라서 시간이 지나면 정보 검색보다 지식 모델링의 가치가 커진다. 이 부분이 이 레포의 핵심 경쟁력이다. ([GitHub][1])

## 6. 사용 시나리오

README는 이 레포의 주요 활용 대상을 다음과 같이 제시한다.

* 기술 문서 작성자
* 연구자
* PM
* 학생
* 장기적으로 정보를 축적하는 일반 사용자 ([GitHub][1])

이들을 공통으로 묶는 패턴은 명확하다. **여러 출처에서 정보가 점진적으로 들어오고, 시간이 지나며 개념 간 연결이 중요해지는 업무**다. 즉, 단발성 Q&A보다 누적형 분석 작업에 맞다. ([GitHub][1])

귀하의 업무 맥락으로 옮기면, 이 패턴은 특히 다음 분야와 잘 맞는다.

* BL/BOE/DO/CI·PL/INS 같은 문서군의 누적 정리
* HS/통관/운임/DEM·DET 규칙의 개념 맵 관리
* 벤더·캐리어·포트·현장별 이슈 히스토리 축적
* 클레임/지연/예외사례의 재발 패턴 분석

이 적용 가능성은 README의 설계 구조에 근거한 해석이다. 원문 유형이 PDF, 보고서, 이메일 텍스트, 경쟁 문서 등 매우 넓게 허용되기 때문에, 물류 문서 아카이브에도 전용 구조로 전환 가능하다. ([GitHub][1])

## 7. 장점 평가

### 7.1 축적형 지식 운영에 강함

가장 큰 장점은 “대화의 결과가 사라지지 않는다”는 점이다. 질문과 분석이 새로운 위키 페이지가 되므로, 팀의 판단 히스토리와 분석 자산이 누적된다. 이는 보고서, 리스크 리뷰, 교훈 회고에 매우 유리하다. ([GitHub][1])

### 7.2 원문과 가공물 분리

`raw/`를 AI가 수정하지 않는 구조는 매우 건전하다. 원문 무결성을 유지하면서도, `wiki/`에서 자유롭게 구조화와 재해석을 수행할 수 있기 때문이다. 감사 추적과 책임 분리에 적합하다. ([GitHub][1])

### 7.3 Obsidian 기반 탐색성

README는 Graph View 사용을 강하게 권장한다. 이는 사용자가 단순 검색 결과를 보는 수준을 넘어, 지식 허브와 고아 페이지를 시각적으로 파악하게 한다. 장기 운영에서는 이런 시각화가 누락 영역과 과잉 집중 영역을 찾는 데 유용하다. ([GitHub][1])

### 7.4 스키마 커스터마이징 가능

`CLAUDE.md`를 수정해 도메인별 페이지 타입을 추가하거나 워크플로를 바꿀 수 있다. 예를 들어 API 엔드포인트, 고객 세그먼트, 레시피 변형 같은 새 개체를 추가할 수 있다고 README는 설명한다. 즉, 이 레포는 완성형 제품이 아니라 **도메인 특화 가능한 지식 운영 프레임워크**다. ([GitHub][1])

## 8. 한계와 리스크

### 8.1 운영 품질이 스키마와 프롬프트 품질에 크게 의존

이 시스템은 데이터베이스 스키마 대신 `CLAUDE.md`라는 자연어 규칙 문서에 크게 의존한다. 즉, 잘 설계하면 강력하지만, 규칙이 모호하면 위키 구조가 쉽게 흔들릴 수 있다. README도 스키마를 직접 수정해 맞춤화하라고 전제한다. ([GitHub][1])

### 8.2 사실성 검증 체계는 약함

README에는 요약, 개체 생성, 링크 생성, lint 절차는 있으나, 엄격한 출처 검증·승인 워크플로·권한 통제·민감정보 마스킹 체계는 명시적으로 보이지 않는다. 따라서 규제·계약·민감 문서 환경에 그대로 투입하기엔 통제 레이어가 부족하다. 이 평가는 README에 드러난 범위 기준의 분석이다. ([GitHub][1])

### 8.3 협업형 운영보다 개인용 패턴에 가까움

README는 Cursor와 Obsidian을 좌우로 띄워 개인이 대화하며 위키를 키우는 흐름을 기본 경험으로 제시한다. 즉, 팀 승인, 롤백, RBAC, 감사로그 중심의 엔터프라이즈 운영모델은 기본 포함이 아니다. ([GitHub][1])

### 8.4 대규모 문서군에서의 유지비용 불명확

소스가 늘어나면 위키가 풍부해지는 장점이 있지만, 동시에 링크 정합성, 중복 개념, 오래된 문서 관리 문제가 생긴다. README는 `lint the wiki`를 해결책으로 제시하지만, 규모 확장 시 성능·정확도·운영 부하가 어디까지 감당되는지는 README만으로는 확인되지 않는다. 이 부분은 ⚠️AMBER: 대규모 실운영 성능은 별도 검증이 필요하다. ([GitHub][1])

## 9. 귀하의 물류·문서 자동화 업무에 적용할 경우

이 레포를 HVDC 프로젝트 물류 관점에서 재해석하면, 핵심 가치는 “문서 저장”이 아니라 “운영 지식의 구조화”다. 예를 들어 아래와 같이 페이지 타입을 재정의할 수 있다.

* Source: CI/PL, BL, BOE, DO, 보험증권, 포트 메모, e-mail thread
* Concept: HS code rule, UAE customs rule, Incoterm, DEM·DET rule
* Persona: Site logistics, broker, vendor, carrier, consignee
* Product 대신 Asset: 프로젝트, site, vessel, shipment lane
* Analysis: delay root cause, exception pattern, customs rejection pattern, cost leakage review

이런 구조라면 “어제 온 메일”보다 “지금까지 누적된 통관 이슈 패턴”을 더 잘 다룰 수 있다. 즉, 단순 검색형 챗봇보다 **운영 판단용 지식 위키**로 적합하다. 이 부분은 README 구조를 기반으로 한 도메인 적용 해석이다. ([GitHub][1])

## 10. 실무 도입 시 필요한 보강 항목

이 레포를 그대로 쓰기보다, 아래 통제 장치를 반드시 추가하는 편이 안전하다.

### 10.1 보안·민감정보 분리

민감 문서 원문을 그대로 LLM ingest 대상으로 넣으면 NDA/PII/단가 유출 리스크가 있다. 따라서 원문층과 요약층을 분리하고, 민감정보 마스킹 또는 비식별화 파이프라인을 먼저 두는 것이 바람직하다. 이 보강 방향은 귀하가 관리 중인 UI/UX 운영 가이드의 **Guide ↔ Materials 분리**, P2 비포함 원칙과 정합적이다. 

### 10.2 승인형 Agentic UX

이 레포는 개인용 생산성 패턴에는 적합하지만, 실운영 도구라면 “실행 전 요약 → 승인/취소 → 결과/근거 → 재시도/롤백” 같은 승인형 UX가 필요하다. 이는 특히 문서 수정, 규정 해석 저장, 대외 보고 초안 생성에 필수다. 귀하의 UI/UX 가이드는 툴 실행을 승인형으로 설계하고 감사 로그를 남기도록 요구한다. 

### 10.3 성능·품질 기준

문서 수가 많아질수록 위키 정합성·렌더링 속도·탐색성이 중요해진다. 이때는 단순 생성보다 운영 품질 기준이 필요하다. 업로드해 둔 Vercel Agent Skills 운영 가이드도 waterfall 제거, bundle size 감소, 감사→계획→수정→검증→회귀방지 흐름을 핵심 운영 방법으로 제시한다. 즉, 이 위키를 프론트엔드 도구로 제품화하려면 그 수준의 성능·검증 체계가 붙어야 한다. 

## 11. 종합 판단

이 레포는 완성형 SaaS가 아니라, **Karpathy식 장기 기억형 지식 위키를 빠르게 실험할 수 있는 템플릿**으로 보는 것이 정확하다. 강점은 원문을 구조화된 위키로 전환하고, 질문 자체를 다시 지식 자산으로 축적한다는 점이다. README가 제시하는 ingest, page types, lint, graph view, schema customization은 모두 그 목적에 일관되게 맞춰져 있다. ([GitHub][1])

반면, 엔터프라이즈 문서 운영에 필요한 승인, 권한, 마스킹, 감사, 규정준수, KPI, 복구 경로는 기본 제공 범위를 넘어선다. 따라서 개인 연구·지식관리에는 즉시 유용하지만, 프로젝트 물류·통관·계약·비용처럼 민감도가 높은 도메인에는 **보안 분리와 승인형 실행 레이어를 얹은 후** 도입하는 것이 적절하다. 이 결론은 README에 드러난 범위와 귀하의 운영 원칙을 결합한 분석이다. ([GitHub][1])

## 12. 권고안

### Option A — 개인 지식 위키 실험용

가장 빠른 방식이다. Cursor + Obsidian 조합으로 바로 돌려보고, `CLAUDE.md`를 물류 문서 타입에 맞게 개조한다. 저비용이지만 보안 통제는 약하다. ([GitHub][1])

### Option B — 팀 내부 비식별 운영 위키

`raw/` 투입 전 마스킹 레이어를 두고, source·concept·analysis 구조를 통관/운송/예외관리 중심으로 재정의한다. 실무 적용성이 가장 높다. ([GitHub][1])

### Option C — 제품화 전제 아키텍처

위키 생성 파이프라인에 승인 UX, 감사 로그, KPI, 접근성, 성능 최적화까지 붙여 내부 도구로 제품화한다. 가장 강력하지만 구현비용이 가장 크다. 이 경우 귀하의 UI/UX 가이드와 Vercel 운영 가이드를 함께 기준 문서로 쓰는 편이 맞다.
